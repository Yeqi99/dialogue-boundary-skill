# English Introduction Draft

Large language models increasingly act as communication mediators. Users ask them not only to answer factual questions, but also to judge whether an idea is worth pursuing, write messages for customers or teachers, generate structured tool instructions, and convert conversations into executable tasks for downstream agents. In these settings, output reliability depends not only on factual correctness, but also on whether the model binds the correct goal, evaluation standard, speaker, recipient, and context visibility boundary.

Many discussions of LLM failure focus on hallucination, reasoning errors, prompt injection, or prompt leakage. In practical use, however, another class of failures appears: the model produces fluent and plausible text, but under the wrong communication boundary. The first subtype occurs in evaluative answers. When a user asks whether an idea is meaningful, valuable, or worth turning into a paper, the model often supplies an implicit evaluation frame. If the model defaults to top-conference standards, complete product requirements, or high-cost experimental rigor while the user's actual goal is a research-oriented open-source artifact, technical report, preprint, or credibility layer, the answer can be misleading despite being factually reasonable. We call this Frame Boundary Failure, or Evaluation Frame Misbinding.

The second subtype occurs in recipient-facing generation. When a model writes text for a customer, HR representative, teacher, public audience, tool, or downstream agent, it may include user-model negotiation, tone corrections, model self-reflection, prompt traces, or explanations meant only for the upstream user. This error is not always prompt leakage in the narrow sense. It is a failure to separate visible context from internal context and forbidden context. We call this Dialogue Boundary Failure.

We propose Boundary Cards as a lightweight intervention for these failures. The method applies two gates before generation. Gate 1, Frame Lock, binds the user goal, evaluator, success standard, and cost model before evaluative answers. Gate 2, Dialogue Boundary Lock, binds the speaker, recipient, output type, visible context, internal context, and forbidden context before recipient-facing generation. After generation, leakage review checks for meta-communication leakage, prompt traces, agent task pollution, and self-correction leakage; Instruction Drift Guard remains a secondary guardrail/check for conceptual distortion, not one of the two primary communication boundary failures.

This work is not a politeness guide, a customer-service template collection, a generic prompt-engineering recipe, or a complex multi-agent communication protocol. It identifies a practical class of LLM communication errors and proposes a reusable low-cost method for reducing them. We further outline Communication Boundary Cases, a dataset design and annotation protocol for evaluating boundary reliability across evaluative answers, third-party messages, agent tasks, tool instructions, and reusable skill generation.
