# 中文 Introduction 初稿

大语言模型正在成为一种通用的沟通中介。用户不再只是要求模型回答事实问题，也会要求模型判断一个想法是否值得做、替自己写给客户或老师的消息、生成给工具的结构化指令，或把当前对话整理成下游 Agent 可以执行的任务。在这些场景中，输出是否可靠不仅取决于事实是否正确，还取决于模型是否正确理解了这段输出服务于什么目标、由谁说出、说给谁听，以及哪些上下文可以被接收方看到。

现有关于大模型错误的讨论常集中在事实幻觉、推理错误、prompt injection 或 prompt leakage。然而，在实际使用中还存在一类更隐蔽的错误：模型生成的文本流畅、合理，甚至事实正确，但绑定了错误的沟通边界。第一类错误发生在评价性回答中。当用户问“这个想法有没有意义”“是否值得写成论文”“项目有没有价值”时，模型往往会自动补全一个评价框架。如果模型默认采用顶会论文、完整产品或高成本实验的标准，而用户真实目标只是形成研究型开源成果、技术报告、预印本或项目可信度材料，那么回答就会偏离用户目标。本文将这类问题称为 Frame Boundary Failure 或 Evaluation Frame Misbinding。

第二类错误发生在面向特定接收方的文本生成中。当模型为客户、HR、老师、公众、工具或下游 Agent 生成文本时，它可能混入用户和模型之间的协商过程、用户刚刚提出的语气纠正、模型的自我反思、提示词痕迹或给用户看的解释。这类错误并不一定泄漏系统提示词，但它泄漏了不属于接收方的沟通层信息。本文将其称为 Dialogue Boundary Failure。

为降低这两类错误，本文提出一种轻量方法：Boundary Card。该方法在生成前执行两个 gate。Gate 1 是 Frame Lock，用于在评价性回答前绑定用户目标、评价者、成功标准和成本模型。Gate 2 是 Dialogue Boundary Lock，用于在生成接收方面向文本前绑定说话人、接收方、输出类型、可见上下文、内部上下文和禁止上下文。生成后，模型再执行 leakage review，检查是否出现元沟通泄漏、提示词痕迹、Agent 任务污染或自我修正泄漏；Instruction Drift Guard 是二级防歪检查项，用于检查概念漂移，不是两类核心沟通边界失效之一。

本文的目标不是提出一个复杂多智能体协议，也不是把问题简化为礼貌话术或普通 prompt 工程。本文试图刻画一种常见但尚未被充分命名的 LLM 沟通错误类型，并给出一个可复用、低成本、便于人工评估的干预方法。我们进一步给出 Communication Boundary Cases 数据集设计、标注指南和实验方案，为后续技术报告、预印本或 workshop paper 提供基础材料。
